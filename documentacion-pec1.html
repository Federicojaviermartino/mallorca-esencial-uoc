1. Introduccion y descripcion del proyecto

El presente documento recoge la documentacion completa del proceso de desarrollo de la PEC1 de la asignatura Herramientas HTML y CSS II. El proyecto consiste en un sitio web de una sola pagina (single-page) dedicado a Valldemossa, Soller y Palma de Mallorca, tres de los destinos mas representativos de la isla, conectados por la Serra de Tramuntana, declarada Patrimonio de la Humanidad por la UNESCO.

El sitio web ha sido desarrollado partiendo de UOC Boilerplate (version 3.13.0) e incorpora un workflow moderno de desarrollo frontend con Parcel como module bundler, Sass como preprocesador de estilos, y un conjunto de dependencias externas que enriquecen la experiencia del usuario. Se ha aplicado la metodologia BEM para la nomenclatura de clases CSS, junto con la guia de estilo de @mdo (Code Guide) para el ordenamiento de propiedades, validado todo ello mediante Stylelint.

La pagina incluye las siguientes secciones: navegacion fija, hero de pantalla completa, seccion descriptiva del destino, tarjetas de puntos destacados, galeria de imagenes, seccion de como llegar con pasos visuales, mapa interactivo con Leaflet y cinco marcadores, consejos practicos para la visita, y footer con informacion del proyecto.

2. Proceso de desarrollo

2.1 Descarga y configuracion inicial

El primer paso fue clonar el repositorio de UOC Boilerplate desde GitHub mediante el comando git clone. A continuacion se verifico que la version de Node.js instalada fuera superior a 20.x ejecutando node -v en el terminal.

git clone https://github.com/uoc-advanced-html-css/uoc-boilerplate.git valldemossa-soller-palma
cd valldemossa-soller-palma
npm install

El comando npm install descargo e instalo todas las dependencias definidas en el package.json dentro de la carpeta node_modules, generando tambien el fichero package-lock.json con las versiones exactas de cada paquete.

2.2 Instalacion de dependencias

Se instalaron las siguientes dependencias adicionales al proyecto:

npm install --save @fortawesome/fontawesome-free
npm install --save leaflet
npm install --save @fontsource/playfair-display @fontsource/source-sans-pro
npm install --save-dev stylelint stylelint-scss stylelint-order
    stylelint-config-recommended-scss stylelint-config-recess-order

Las dependencias de produccion (--save) se anadieron como dependencies en el package.json, mientras que las herramientas de desarrollo (--save-dev) se anadieron como devDependencies. Esta separacion es importante porque las devDependencies solo se necesitan durante el proceso de desarrollo y compilacion, no en el bundle final.

2.3 Entorno de desarrollo

Durante el desarrollo se utilizo el comando npm run dev, que ejecuta las tareas clean y parcel:serve de forma secuencial. Parcel lanza un servidor de desarrollo en http://localhost:8123 que monitoriza los cambios en los ficheros fuente y actualiza el navegador automaticamente (hot reload). Esto permite un flujo de trabajo agil donde los cambios en HTML, SCSS o JavaScript se reflejan al instante.

2.4 Compilacion para produccion

Para la compilacion final se utilizo el comando npm run build, que ejecuta secuencialmente: clean (limpia carpetas de cache y dist), stylelint (valida las hojas de estilos) y parcel:build (compila, concatena, minifica y optimiza todos los assets). El resultado se genera en la carpeta dist/ con ficheros optimizados para produccion, incluyendo la conversion automatica de imagenes a formato WebP, la minificacion de CSS y JavaScript, y el hashing de nombres de ficheros para cache busting.

npm run build

3. Eleccion y justificacion de la metodologia

3.1 Metodologia elegida: BEM

Se ha elegido BEM (Block, Element, Modifier) como metodologia principal para la nomenclatura de clases CSS. BEM propone una convencion de nombres estructurada que divide los componentes en tres categorias: bloques (componentes independientes), elementos (partes de un bloque que no tienen sentido por si solas) y modificadores (variaciones de un bloque o elemento).

La nomenclatura sigue el patron: .bloque__elemento--modificador, donde los dos guiones bajos separan el bloque del elemento y los dos guiones separan el bloque o elemento del modificador.

3.2 Justificacion de la eleccion

La decision de usar BEM se basa en los siguientes criterios:

  • Claridad: la nomenclatura BEM hace explicita la relacion entre los elementos HTML y sus estilos. Al leer una clase como .hero__title, queda claro que se trata del titulo dentro del bloque hero.
  • Especificidad plana: BEM fomenta el uso de selectores de una sola clase, evitando problemas de especificidad que surgen con selectores anidados o combinados.
  • Modularidad: cada bloque es un componente independiente que puede reutilizarse sin conflictos. Los bloques definidos en este proyecto (nav, hero, about, highlights, gallery, route, map-section, tips, footer) son autosuficientes.
  • Escalabilidad: aunque este proyecto es de una sola pagina, la nomenclatura BEM facilita la expansion futura sin riesgo de colisiones de nombres.
  • Compatibilidad con Sass: la sintaxis de anidamiento de Sass con el operador & se integra perfectamente con BEM, permitiendo escribir los estilos de forma compacta y organizada.

3.3 Guia de estilo complementaria: Code Guide (@mdo)

Ademas de BEM, se ha adoptado la guia de estilo de Code Guide de Mark Otto para el ordenamiento de las propiedades CSS. Esta guia propone un orden logico que agrupa las propiedades por tipo: primero las de posicionamiento (position, top, right...), despues las del modelo de caja (display, width, padding, margin...), luego las tipograficas (font-family, font-size, color...) y finalmente las visuales (background, border, box-shadow...). Este orden se aplica automaticamente mediante el plugin stylelint-config-recess-order.

3.4 Aplicacion practica de BEM en el proyecto

Cada seccion de la pagina esta definida como un bloque BEM independiente. Por ejemplo, el bloque .nav contiene los elementos .nav__container, .nav__brand, .nav__toggle, .nav__list, .nav__item y .nav__link. El modificador .nav__list--open se usa para controlar la visibilidad del menu en dispositivos moviles.

Otro ejemplo es el bloque .highlights, donde .highlights__card es un elemento y podria tener un modificador como .highlights__card--featured si se quisiera destacar alguna tarjeta en particular. Esta estructura se repite de forma consistente en todos los bloques del proyecto.

4. Configuracion de Stylelint

Stylelint se ha configurado en el fichero .stylelintrc de la raiz del proyecto. Se extienden dos configuraciones base: stylelint-config-recommended-scss, que proporciona un conjunto de reglas recomendadas para ficheros SCSS, y stylelint-config-recess-order, que aplica el orden de propiedades segun la guia de estilo de @mdo. Este ultimo depende del plugin stylelint-order, que se instalo como devDependency adicional para gestionar el ordenamiento automatico de las propiedades CSS.

Las reglas personalizadas incluyen:

  • selector-class-pattern: expresion regular que valida que todas las clases sigan la nomenclatura BEM. El patron acepta bloques (.bloque), elementos (.bloque__elemento) y hasta dos modificadores (.bloque--mod1--mod2).
  • max-nesting-depth: 4 - limita la profundidad maxima de anidamiento en Sass a 4 niveles para evitar selectores excesivamente especificos.
  • scss/at-rule-no-unknown: true - valida que las at-rules utilizadas en SCSS sean conocidas, previniendo errores tipograficos.
  • declaration-block-no-duplicate-properties: true - previene propiedades duplicadas dentro de un mismo bloque de declaraciones.
  • color-no-invalid-hex: true - asegura que los codigos hexadecimales de color sean validos.
  • font-family-no-duplicate-names: true - evita la repeticion de nombres de fuentes en una misma declaracion.
  • block-no-empty: true - senala bloques de declaraciones vacios que deben eliminarse.
  • no-empty-source: true - detecta ficheros de estilos vacios.

Ademas, se anadio el script de Stylelint al package.json y se integro en la rutina de build, de forma que la compilacion para produccion solo se completa si las hojas de estilos pasan la validacion. El comando npm run build ejecuta secuencialmente: clean, stylelint y parcel:build.

5. Caracteristicas de Sass utilizadas

Se han utilizado todas las caracteristicas de Sass requeridas por el enunciado:

5.1 Variables

Definidas en el fichero _variables.scss, se han creado variables para la paleta de colores (inspirada en el Mediterraneo: azules, turquesas, arena), tipografia (familias de fuentes, tamanos, alturas de linea), espaciado (sistema de 8 puntos con valores de xs a 3xl), breakpoints para responsive design, sombras, bordes redondeados y duraciones de transicion. Ademas, se han usado funciones nativas de Sass como color.scale() para generar colores derivados.

5.2 Anidado (Nesting)

El anidado se ha utilizado extensivamente en combinacion con BEM, usando el operador & de Sass para generar los selectores de elementos y modificadores. Por ejemplo, dentro del bloque .nav, los elementos se definen como &__container, &__brand, &__list, etc. Tambien se ha usado el anidado para las media queries con @include respond-to(), manteniendo las reglas responsive junto a las reglas base de cada elemento.

5.3 Funciones personalizadas

Se han creado tres funciones personalizadas en el fichero _mixins.scss:

  • to-rem($px): convierte valores en pixeles a unidades rem, facilitando el trabajo con unidades relativas. Utiliza el modulo sass:math para la division.
  • overlay-color($base-color, $opacity): genera una version semi-transparente de un color mediante rgba, utilizada para los overlays del hero y la navegacion.
  • text-contrast($bg): devuelve un color de texto claro u oscuro segun la luminosidad del color de fondo, usando color.channel() del modulo sass:color. Se aplica para garantizar la legibilidad del texto sobre fondos de diferentes tonalidades.

5.4 Parciales e importacion

Los estilos estan organizados en multiples ficheros parciales (prefijo _) que se importan mediante la directiva @use de Sass. La estructura es la siguiente: _variables.scss (variables y colores), _mixins.scss (funciones y mixins), _dependencies.scss (dependencias de npm) y layouts/_home.scss (estilos de la pagina). El punto de entrada es main.scss, que importa los demas parciales. El uso de @use en lugar de @import (deprecated) permite un control mas preciso de los namespaces y evita la contaminacion del scope global.

5.5 Mixins

Se han definido los siguientes mixins reutilizables:

  • respond-to($breakpoint): encapsula las media queries para los breakpoints definidos (sm, md, lg, xl), facilitando el enfoque mobile first.
  • container: aplica un ancho maximo, margenes automaticos y padding lateral para centrar el contenido.
  • section-padding: aplica padding vertical consistente a las secciones, con valores diferentes segun el breakpoint.
  • hover-lift($scale): efecto visual de elevacion al hacer hover, con transicion de transform y box-shadow.
  • heading($size): aplica la fuente serif (Playfair Display), peso y altura de linea para titulos.
  • sr-only: oculta visualmente un elemento pero lo mantiene accesible para lectores de pantalla.

7. Decisiones de diseno y desarrollo

7.1 Paleta de colores

La paleta de colores se inspira en los elementos naturales de Mallorca: los azules profundos del Mediterraneo ($color-sea-deep: #1a3a5c), los turquesas de las aguas poco profundas ($color-turquoise: #48c9b0), los tonos arena de la costa ($color-sand: #f5e6c8) y los marrones de la piedra de Valldemossa ($color-rock: #6b5b4f). Esta paleta crea una atmosfera coherente con el destino que se presenta.

7.2 Tipografia

Se ha elegido un emparejamiento clasico: Playfair Display (serif) para titulos y encabezados, que aporta elegancia y personalidad, y Source Sans Pro (sans-serif) para el cuerpo de texto, que ofrece una lectura comoda en pantalla. Ambas fuentes se cargan localmente mediante Fontsource.

7.3 Estructura de la pagina

La pagina sigue un flujo vertical narrativo que guia al visitante desde la primera impresion (hero a pantalla completa) hasta la informacion practica (consejos y mapa). Las secciones alternan fondos claros y oscuros para crear un ritmo visual que mantiene el interes del usuario. La seccion "Como llegar" usa un fondo con gradiente en tonos azules que rompe con el patron claro y aporta variedad.

7.4 HTML semantico

Se ha prestado especial atencion a la semantica del HTML. La estructura del documento utiliza las etiquetas <header> para envolver la navegacion, <main> para el contenido principal de la pagina, <section> para cada bloque tematico, <article> para las tarjetas de puntos destacados, <figure> y <figcaption> para las imagenes de la galeria, y <footer> para el pie de pagina. Todos los estilos se aplican exclusivamente desde las hojas de estilos SCSS, sin recurrir a estilos inline en el HTML.

7.5 PostHTML

Se ha utilizado el plugin posthtml-include, configurado en el fichero .posthtmlrc, para separar el footer en un fichero parcial (views/footer.html). La etiqueta no estandar include permite incluir este fichero en el index.html durante la compilacion, manteniendo el codigo organizado y facilitando la reutilizacion del footer si se anadieran mas paginas en el futuro.

7.6 JavaScript

El fichero main.js contiene tres funcionalidades: la inicializacion del mapa Leaflet con cinco marcadores geolocalizados (Valldemossa, Soller, Port de Soller, Catedral de La Seu y Castell de Bellver) con popups informativos; el toggle del menu de navegacion en dispositivos moviles con gestion de estados ARIA para accesibilidad; y el smooth scroll para los enlaces internos de navegacion.

8. Responsive design

Se ha seguido una aproximacion mobile first, donde los estilos base estan pensados para pantallas pequenas y se van adaptando a pantallas mas grandes mediante media queries progresivas. Los breakpoints definidos son: sm (576px), md (768px), lg (992px) y xl (1200px).

Ejemplos concretos de adaptacion responsive:

  • Navegacion: en movil se muestra un boton hamburguesa que despliega el menu verticalmente. En pantallas medianas (md) la navegacion se muestra horizontalmente.
  • Hero: el tamano del titulo se escala de 48px (movil) a 64px (desktop) y el subtitulo de 18px a 24px.
  • Highlights: las tarjetas pasan de 1 columna (movil) a 2 columnas (sm) y 3 columnas (lg).
  • Galeria: de 1 columna a 2 (sm) y 4 columnas (lg).
  • Seccion "Como llegar": de 1 columna a 2 (md) y 4 (lg).
  • Mapa: la altura aumenta de 350px a 450px en pantallas medianas.
  • Secciones: el padding vertical aumenta de 4rem a 6rem en pantallas medianas.

10. Conclusiones

El desarrollo de esta PEC ha permitido consolidar los conocimientos adquiridos en los modulos 1 y 2 de la asignatura, poniendo en practica un workflow moderno de desarrollo frontend. El uso de UOC Boilerplate como punto de partida facilito la configuracion inicial del proyecto, permitiendo centrar el esfuerzo en el desarrollo y el diseno.

La aplicacion de BEM como metodologia de nomenclatura, combinada con la guia de estilo de @mdo para el orden de propiedades y la validacion automatica con Stylelint, ha resultado en un codigo CSS ordenado, coherente y facil de mantener. Las funcionalidades de Sass (variables, anidado, funciones, parciales, mixins) han demostrado su utilidad para generar hojas de estilos mas eficientes y modulares.

La incorporacion de dependencias externas como Leaflet, FontAwesome y Fontsource ha enriquecido la experiencia del usuario, aportando interactividad (mapa con cinco puntos de interes entre Valldemossa, Soller y Palma), iconografia (iconos) y personalidad tipografica (fuentes), todo ello gestionado de forma local a traves de npm sin depender de CDNs externos.
